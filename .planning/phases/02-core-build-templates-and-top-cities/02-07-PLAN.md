  <action>
    Replace skeleton with a universal content detail template that handles multiple content types. Server Component.

    **Content type resolution strategy:**
    Since blog/guide, comparison, and case study all route to /resources/[slug]/, the page must try multiple queries to find the content:
    1. First try BLOG_GUIDE_QUERY (most common)
    2. If null, try COMPARISON_QUERY
    3. If null, try CASE_STUDY_QUERY
    4. If all null, notFound()

    Alternative (more efficient): Create a UNIVERSAL_RESOURCE_QUERY in this file using GROQ union:
    ```
    *[slug.current == $slug && _type in ["blogGuide", "comparison", "caseStudy"]][0] {
      _type,
      title, slug, seo, publishing,
      // blogGuide fields
      _type == "blogGuide" => { category, body, excerpt, heroImage, author, "cityTitle": city->title, "serviceTitle": service->title },
      // comparison fields
      _type == "comparison" => { competitors, body, "serviceTitle": service->title },
      // caseStudy fields
      _type == "caseStudy" => { clientName, outcome, body, heroImage, "cityTitle": city->title },
    }
    ```
    Define this query inline in the file (or add to a queries file).

    **generateStaticParams:** Combine all three list queries to return all slugs.

    **generateMetadata:** Use the fetched document's seo fields via generatePageMetadata. Path: `/resources/${slug}`. Fallback title based on _type.

    **JSON-LD:** For all types, use buildArticleSchema:
    - headline: title
    - author: author || "MoveSmart Rentals"
    - datePublished: publishing.publishedAt
    - dateModified: publishing.updatedAt || _updatedAt
    - description: seo.metaDescription || excerpt || outcome

    **Template rendering by type:**

    All types get:

    2. Article header: h1 title, metadata bar (author, date, category badge, city if applicable)
